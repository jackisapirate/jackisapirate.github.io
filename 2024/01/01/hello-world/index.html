Self Introduction

Hi, my name is Todd, and I have eight years of working experience as a Full Stack Developer. I am familiar with building scalable web applications and data management systems. I’ve worked across several domains including retail, insurance, bank, and e-commerce.

I have experience with Java (FrameWork) including Spring Boot, Spring MVC, Spring JPA, Hibernate, Mybatis, Spring Security with OAuth. I’ve also worked with messaging queue like Kafka to build asynchronous communication pipelines, using Restful API to do synchronous communication between services.

On the data side, I’m comfortable working with both SQL and NoSQL databases, including MySQL, MongoDB, Redis, and Elasticsearch. I also have strong experience with CI/CD pipelines using Jenkins, Kubernetes, and dependency management tools like Maven and Gradle. Additionally, I’ve worked with AWS services such as EC2 and RDS.

I am experienced the full software development lifecycle and have worked in both Waterfall and Agile-Scrum environments. I also practiced Test-Driven Development and have used tools like JUnit, Mockito, Selenium, Cucumber, and TestNG for automation test.

Most recently, I worked on Target’s RMS (Return Management System) team, where our team focused on improving product return operations for customers, merchants, and in-store associates. My work included developing and enhancing return-related features, writing automation tests, deploying services into testing environments, and performing end-to-end validation. These contributions helped improve testing efficiency, system stability, and the overall return experience for end users.

When customers initiated a return, the order information was sent to our system through Kafka. Our service processed the data, generated shipping labels, issued refunds when applicable, and continuously tracked the return shipment status. Once the updates were available, we published the processed information to downstream services through Kafka.

On the other side of the workflow, merchants could also submit disputes, which were received by our service through Kafka. We compared the dispute data against the return records, tracked the status, and sent the results to downstream services. The system supported manual review for merchant disputes, and based on the final decision, we either approved or rejected the refund.

当customer发送请求，在前端点击return某个商品，前端会发送http请求到后端。这个请求首先通过api gateway，（这里说一下api gateway的作用）通过load balancer（说一下load balance的作用），然后进入我们退货的service named RMS。我们实际部署用的k8s，如果流量过大会弹性增减RMS services。请求会发送到对应的endpoints，首先在数据库里增加一条退货信息，然后去Order Service里查询订单信息，然后根据返回的订单状态决策是否需要退款，是否需要顾客退还商品。例如如果订单还未邮寄给顾客就不需要顾客退还商品。如果顾客已经收到订单，就需要顾客退还商品。我们的RMS经过处理后，会通知下游service，给下游service 发送Kafka message，例如通知Payment Service 给顾客refund，通知Delivery Service给顾客打印快递单子等。

When a customer initiates a return request on the frontend, the application sends an HTTP request to our backend. The request was first passed through the API Gateway, which handles responsibilities such as authentication, rate limiting, and routing. It then goes through the Load Balancer, which distributes incoming traffic evenly across service instances to ensure high availability and prevent overload.

After that, the request reaches our return-management service, called RMS. The service is deployed on a Kubernetes cluster, so it can automatically scale up or down depending on traffic volume.

Inside RMS, the request is routed to the corresponding endpoint. We first create a new return record in the database, then call the Order Service to fetch order details. Based on the returned order status, RMS decides whether the customer should be refunded and whether the item needs to be physically returned. For example, if the order has not yet been shipped, the customer does not need to send anything back. If the order has already been delivered, we require the customer to return the item.

After RMS finishes processing, it notifies downstream services by publishing Kafka messages. For instance, we send events to the Payment Service to trigger the refund, and to the Delivery Service so they can generate a return shipping label for the customer.

Customer submits return → HTTP request → API Gateway (auth, rate limiting) → Load Balancer → RMS → Input validation & enrichment → check cache → create return record → call Order Service & Fraud Detection Service → decide refund / physical return → enqueue async tasks → publish Kafka messages → downstream services (Payment Service, Delivery Service)

🎯 1. 关于整体架构

  1. 为什么要同时使用 API Gateway 和 Load Balancer？它们的职责怎么划分？有没有重叠？

  2. API Gateway 的认证逻辑是如何实现的？你们使用的是 JWT、OAuth2、Session 还是内部 token？为什么？

  3. 你们是基于什么来决定 RMS 要水平扩缩容？是 CPU？队列长度？还是自定义指标？

    在 Kubernetes 环境中，我们其实不需要显式配置传统的 Load Balancer。

    来自外部的流量进入系统，通常会经过两个层级：

    (1) API Gateway（负责逻辑上的网关能力）

    • 请求路由
    • 身份认证 / 鉴权
    • 限流
    • 灰度发布
    • 日志收集
    • 加解密

    云 Load Balancer（Cloud LB）

    • 本质：云厂商提供的负载均衡服务（AWS ELB, GCP LB, Azure LB）
    • 作用：
      • 把外部互联网流量分发到你的 Kubernetes 集群节点上（Node）
      • 保证高可用：如果某个节点挂掉，流量自动导向其他节点
      • 提供固定公网 IP / DNS
    • 触发方式：
      • 当你在 Kubernetes 中创建 Service type=LoadBalancer 时，K8S 会自动向云厂商请求一个 LB，并把它和 Ingress / Service 绑定

    Ingress Controller

    • 本质：Kubernetes 内部的一个组件，用来处理 HTTP/HTTPS 请求的路由。
    • 作用：
      • 根据 URL / Host / Path 把请求转发到对应的 Service（也就是你的 RMS 或其他微服务）
      • 支持 TLS/HTTPS 终止
      • 支持负载均衡（把请求在同一个 Service 的多个 Pod 间分发）
      • 支持限流、重写 URL、认证等高级功能（具体看实现）
    • 常见实现：
      • Nginx Ingress Controller
      • Traefik
      • Istio / Kong（带更多服务网格能力）

    (2) K8S Service（负责负载均衡，但属于基础设施能力）

    • Kubernetes 的 Service（ClusterIP / NodePort / Ingress）会自动将流量均匀分发到多个 RMS pod
    • 这里的负载均衡不需要业务开发参与

    两者没有功能重叠：

    • API Gateway 解决上层业务网关问题
    • K8S 自带的负载均衡解决下层流量分发问题

    所以虽然我之前描述里写成 Load Balancer，但在真实架构里，这个 LB 是 Kubernetes 内置的，而不是我们自己配置的独立组件。

    用户登录时会向 Login Service 发送用户名密码。

    Login Service 负责：

    1. 校验账号
    2. 生成 JWT 或 session token
    3. 返回给前端

    后续 RMS 收到用户请求时：

    • RMS 不会自己解析和验证 token
    • 而是将 token 交给 Login Service 进行统一验证

    原因有两个：

    ① 保证 Token 的中心化控制（Single Source of Truth）

    • 如果 token 格式或规则更新，只需要改 Login Service
    • 不需要每个微服务更新

    ② 避免业务服务中嵌入安全逻辑，降低耦合

    • RMS 专注业务逻辑（Return Management）
    • 安全逻辑由 Login Service 专门负责

    多数公司会使用这种“Authentication Service + API Gateway + 微服务”的组合方式来保证一致性、安全性和可维护性。

    在我们的部署中，RMS 是通过 Kubernetes 的 HPA（Horizontal Pod Autoscaler）进行自动扩缩容的。

    虽然具体的监控指标由平台团队配置，但一般会基于以下几类指标：

    ① CPU 使用率（最常见）

    • 比如 pod 的 CPU 连续 1～2 分钟超过 70% 就扩容

    ② 内存使用率

    • 防止应用频繁 OOM

    ③ 自定义业务指标（更高级）

    • 请求响应时间
    • Pending 请求数量
    • Kafka 消费速率

    ④ 事件队列长度

    • 如果消息堆积，说明处理能力不足，需要扩容

    我们团队不需要手动扩容，只要 RMS 流量上升，K8S 会自动启动更多 pod；当流量下降时，pod 会自动减少。

📡 2. 服务之间的通信 & 性能

  1. RMS 调用 Order Service 时，是同步调用还是异步调用？为什么？

  2. 如果 Order Service 挂掉了，RMS 怎么处理？你们有用重试、fallback、熔断吗？

  3. 你们如何保证 Order Service 返回的数据不会拖慢整个链路？有做缓存吗？缓存策略是什么？

    RMS 调用 Order Service 是同步的 REST 调用。
    原因是：RMS 在决定是否允许退款、是否需要用户寄回商品之前，必须立即拿到订单的当前状态（如 pending, shipped, delivered）。

    这些信息是决策链路的一部分，因此必须同步获取，否则 RMS 无法继续业务流程。

    另外，订单状态属于用户级别的关键数据，不适合做异步查询，否则会导致用户端体验不一致或状态过期。

    总结一句话：
    Order Service 是关键业务依赖，必须实时获取状态 → 所以采用同步调用。

    我们在 RMS 与 Order Service 的调用链路上使用了 Circuit Breaker（熔断器），并且带有 超时控制和自动重试。

    调用失败时，处理策略通常是：

    第一步：快速失败（Fail Fast）

    • 如果 Order Service 已经进入熔断状态
    • RMS 不会继续阻塞，而是立即返回错误

    第二步：返回可控的 fallback 结果

    • 比如返回一个 “Order temporarily unavailable” 的业务错误
    • 前端可以提示用户稍后重试

    第三步：自动重试机制

    • 在半开状态下试探性发送请求
    • 一旦 Order Service 恢复，熔断器自动关闭

    这样即避免了整个系统被拖垮，又能在 Order Service 恢复后自动恢复流量。

    总结一句话：
    我们使用 Circuit Breaker + Timeout + Retry 的组合，确保系统在下游不可用时仍能保持稳定性和可恢复性。

    为了避免 Order Service 的响应时间影响 RMS 的整体延迟，我们采用了两类优化手段：

    ① Timeout + Bulkhead（舱壁）隔离

    • 给 Order Service 调用设置严格的超时（例如 300～500ms）
    • 采用线程池隔离，防止 Order Service 调用把 RMS 的线程耗尽

    ② 缓存（Redis）针对“不敏感数据”进行加速
    订单的关键状态不能缓存，但一些不影响实时性的内容可以缓存，例如：

    • SKU 基础信息
    • 店铺/仓库配置
    • 物流方式
    • 用户退货偏好

    缓存策略：

    • 常用 TTL：5 分钟～30 分钟
      （保证数据足够新，又能降低后端压力）
    • Cache Aside 模式（最常见）
      - 先查 Redis
      - Redis miss 再查后端
      - 查完写回 Redis

    对于极关键的实时数据（如 order status），我们不做缓存，而是保证同步调用 + 超时保护。

    总结一句话：
    敏感数据同步查，非敏感数据缓存；超时 + 隔离保证整体链路不会被拖慢。
